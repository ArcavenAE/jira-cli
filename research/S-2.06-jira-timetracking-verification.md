# Verification Report: S-2.06 Worklog Timetracking Claims

**Story:** `.factory/stories/wave-2/S-2.06-worklog-duration-and-cmdb-cache-tuple.md`
**Date:** 2026-05-08
**Producer:** research-agent
**Original verdict:** BLOCK — story spec has multiple factual errors that would cause implementation failure
**User decision:** Option 1 (string passthrough; matches `ankitpokhrel/jira-cli` pattern)

---

## Executive Summary

The story's Part A is technically incorrect on three counts: (1) the endpoint URL it cites returns the time tracking provider, not the working hours config — the correct endpoint is `/rest/api/3/configuration/timetracking/options`; (2) the claimed JSON field names `hoursPerDay`/`daysPerWeek` are wrong — the documented fields are `workingHoursPerDay`/`workingDaysPerWeek` and they are floats, not integers (e.g., 7.6, 5.5); (3) both endpoints require Administer Jira global permission, so for the typical `jr` user this will return 403 — making the "fetch and cache" approach impractical for the majority of users. Additionally, the worklog POST API accepts a `timeSpent` string that Jira parses server-side using its own configured working hours — eliminating the need for client-side duration math entirely.

---

## Per-Claim Verification

### Claim 1 — Endpoint URL and shape

**Claim (verbatim):** `GET /rest/api/3/configuration/timetracking` returns JSON `{"hoursPerDay": N, "daysPerWeek": N}`.

**Verified:** NO
**Confidence:** HIGH

`GET /rest/api/3/configuration/timetracking` exists, but returns the **selected time tracking provider** (`TimeTrackingProvider` object with `key` and `name`), not the working hours configuration. The correct endpoint is `GET /rest/api/3/configuration/timetracking/options`, which returns:

```json
{
  "workingHoursPerDay": 7.6,
  "workingDaysPerWeek": 5.5,
  "timeFormat": "pretty",
  "defaultUnit": "hour"
}
```

Field names are `workingHoursPerDay` / `workingDaysPerWeek` (NOT `hoursPerDay` / `daysPerWeek`), documented as `number` (float), not integer.

### Claim 2 — Auth/scope requirements

**Claim:** `jr` (with standard 3LO scopes) can call the timetracking endpoint and receive 200.

**Verified:** NO
**Confidence:** HIGH

Both `GET /rest/api/3/configuration/timetracking` and `GET /rest/api/3/configuration/timetracking/options` require **"Administer Jira" global permission**. OAuth scope `manage:jira-configuration` is necessary but NOT sufficient — non-admin users will receive 403 even with the scope granted.

### Claim 3 — Worklog POST semantics

**Claim:** `timeSpentSeconds = 7 * 3600 = 25200` is what's sent when `hoursPerDay = 7` and the user enters `"1d"`.

**Verified:** PARTIAL
**Confidence:** HIGH

`timeSpentSeconds` is interpreted as **literal seconds**. The server stores the value verbatim and uses the configured `workingHoursPerDay` only for display formatting. **Critical missed alternative:** The worklog POST also accepts a `timeSpent` string field (e.g., `"1d 2h"`), which Jira parses **server-side** using the instance's configured `workingHoursPerDay`/`workingDaysPerWeek`. This eliminates the need for client-side parsing and admin-only config fetches.

### Claim 4 — Caching policy

**Verified:** PARTIAL
**Confidence:** MEDIUM

7-day TTL is fine for static config data, BUT — the realistic outcome of a 403 means negative-result caching is needed (a shorter TTL like 24h to avoid retrying the admin-only endpoint on every command).

### Claim 5 — Cross-check against `ankitpokhrel/jira-cli` (Go)

**Verified:** YES
**Confidence:** HIGH

`ankitpokhrel/jira-cli` (the most-used Go Jira CLI) forwards the user's `"2d 1h 30m"` string straight to `client.AddIssueWorklog(...)` as a string field. There is no client-side duration parser, no timetracking config fetch, no hardcoded `hoursPerDay`/`daysPerWeek` constants. It is correct on all instances by construction, with zero API calls and zero admin-permission requirements.

---

## Decision

**Option 1 chosen.** Switch from `timeSpentSeconds` (parsed client-side) to `timeSpent` (string, server-parsed). Eliminates timetracking-config fetch, cache, fallback, and admin-permission concerns. Strict regression of `parse_duration` from "calculator" to "validator only".
