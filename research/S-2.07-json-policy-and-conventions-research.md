# S-2.07 Verification: JSON Output Policy and Test Naming Conventions

## Executive Summary

S-2.07 is **mostly verifiable and ready to PROCEED with three concrete corrections**. (1) AC-002's mocked
"token-refresh endpoint" is **not currently testable end-to-end**: the production `jr auth refresh` handler
DOES NOT call `refresh_oauth_token` — it clears keychain entries and re-runs the full OAuth login flow
(`login_oauth → oauth_login`), so a `wiremock` of `POST https://auth.atlassian.com/oauth/token`
intercepts the wrong call. (2) AC-005 must document **`changed: bool`** for `issue move` (verified in
`src/cli/issue/json_output.rs:4-10`) — the spec's "transitioned" alternative is incorrect. (3) The
existing JSON output payload for `auth refresh` (already implemented at `src/cli/auth.rs:808-814` and
`923-932`) uses **`{"status": "refreshed", "auth_method", "next_step"}`**, NOT the
`{"profile", "action", "ok"}` shape proposed in NFR-O-F. The story must reconcile this conflict before
implementation: either keep the existing shape (preferred, consistent with already-shipped code) or
explicitly note the breaking change. The verb-aligned naming policy (NFR-O-J) is supported by industry
precedent (kubectl uses verb-aligned `created/configured/deleted`; AWS uses verb-aligned
`TerminatingInstances/StoppingInstances`), so the "DELIBERATE" framing is **defensible**, not idiosyncratic.

## Per-Claim Verification

### Claim 1 — Atlassian OAuth token refresh endpoint shape

**Claim:** AC-002 mocks "a token-refresh endpoint" without specifying URL or shape. Verify endpoint URL,
request body, response body, expires_in lifetime, and refresh-token rotation behavior.

#### 1a — URL

- **Verified: YES, Confidence: HIGH**
- **Evidence:** `https://auth.atlassian.com/oauth/token` (POST). Confirmed by Atlassian's
  ["Implementing the Refresh Token Flow"](https://developer.atlassian.com/cloud/oauth/getting-started/refresh-tokens/)
  documentation, fetched 2026-05-08.
- **Implication:** Wiremock target URL for AC-002 (if AC-002 is restructured per Claim 2) must be
  exactly `/oauth/token` on the mock host (since `JR_BASE_URL` overrides only the Jira instance, not
  the auth host — see implication for Claim 2).

#### 1b — Request body shape

- **Verified: YES, Confidence: HIGH**
- **Evidence:** Atlassian uses **JSON-encoded** body (NOT form-encoded as RFC 6749 §6 prescribes):
  ```json
  {
    "grant_type": "refresh_token",
    "client_id": "...",
    "client_secret": "...",
    "refresh_token": "..."
  }
  ```
  Source: [Atlassian "Implementing the Refresh Token Flow"](https://developer.atlassian.com/cloud/oauth/getting-started/refresh-tokens/).
- **Implication:** This is a non-standard deviation from RFC 6749 — important if AC-002 mocks the
  endpoint, because matching against form-encoded bodies (which is the OAuth-spec default many
  libraries default to) will silently fail to match. The wiremock matcher must be JSON-aware.

#### 1c — Response body shape

- **Verified: YES, Confidence: HIGH**
- **Evidence:** `{ "access_token", "refresh_token", "expires_in", "scope" }`. Note `token_type` is
  documented in some Atlassian sources but not all; treat as optional. The crucial fact for AC-002
  snapshot stability is that the response **always returns a NEW refresh_token** that supersedes the
  one used for the request (the old one is invalidated). Source:
  [Atlassian docs](https://developer.atlassian.com/cloud/oauth/getting-started/refresh-tokens/).
- **Implication:** Any snapshot test of refresh-flow output must NOT include the actual access/refresh
  token values (they are random). Use `insta` redactions like `".access_token" => "[redacted]"` (see
  Claim 5).

#### 1d — `expires_in` value

- **Verified: YES, Confidence: HIGH**
- **Evidence:** Access tokens live for **3600 seconds (1 hour)** per Atlassian community + docs. The
  Atlassian-issued refresh token rotates on every use; rotating refresh tokens are individually valid
  for 90 days (each new one resets the 90-day window).
- **Implication:** Not directly load-bearing for S-2.07, but lets AC-002 mock pick a realistic value
  (3600) so any future test that asserts on lifetime stays plausible.

#### 1e — Refresh-token rotation

- **Verified: YES, Confidence: HIGH**
- **Evidence:** Quote from Atlassian: "Every new refresh token returned invalidates the refresh token
  used to get the new access token, and your code should replace the existing refresh token with the
  new refresh token." This is a rotating-refresh-token model.
- **Implication:** S-2.07's AC-002 snapshot expectations (the JSON output `{"profile", "action": "refresh", "ok": true}`)
  are NOT affected by rotation since the proposed JSON envelope doesn't include token values. However,
  the underlying production code (`src/api/auth.rs::store_oauth_tokens`) MUST overwrite both
  access AND refresh in the keychain — if a future regression were to write only the new access while
  preserving the old (now-invalid) refresh, the next refresh would fail. This is a property worth
  asserting in the test setup but is OUT OF SCOPE for AC-002 as currently written.

### Claim 2 — `refresh_oauth_token` callability and `jr auth refresh` semantics

**Claim:** AC-002 tests `jr auth refresh --output json` end-to-end with a mocked token-refresh endpoint.
This requires the code path to be invokable today — does it call `refresh_oauth_token`?

#### 2a — Is `jr auth refresh` wired up?

- **Verified: YES, Confidence: HIGH**
- **Evidence:** `src/cli/mod.rs:236-257` defines `AuthCommand::Refresh { profile, oauth, email, token, client_id, client_secret }`.
  `src/cli/auth.rs:845-935` defines `pub async fn refresh_credentials(args: RefreshArgs<'_>)`.
- **Conclusion:** `jr auth refresh` is invokable.

#### 2b — Does `jr auth refresh` call `refresh_oauth_token`?

- **Verified: NO, Confidence: HIGH**
- **Evidence:** `src/cli/auth.rs:895-909` shows `refresh_credentials` dispatches:
  ```rust
  match flow {
      AuthFlow::OAuth => auth::clear_profile_creds(&target)?,  // wipe namespaced oauth-*
      AuthFlow::Token => auth::clear_all_credentials(&[target.as_str()])?,  // wipe shared creds
  }
  let login_result = match flow {
      AuthFlow::Token => login_token(...).await,
      AuthFlow::OAuth => login_oauth(...).await,  // FULL OAuth 3LO browser flow
  };
  ```
  This is a "wipe-then-relogin" pattern — designed for the **macOS Keychain ACL recovery** path
  documented in `src/cli/mod.rs:230-235` (issue #207). It does NOT exercise `refresh_oauth_token`
  at all. The `login_oauth` call goes through the FULL OAuth 3LO authorization-code flow (browser
  redirect, callback listener on port 53682, etc.).
- **Sibling evidence:** A separate function `crate::api::auth::refresh_oauth_token` is referenced
  in `src/cli/auth.rs:791` ("the *future* source — same resolver as `refresh_oauth_token`") and per
  CLAUDE.md the function "resolves credentials internally — callers pass only `profile`. Do not
  re-introduce `client_id`/`client_secret` parameters." This confirms `refresh_oauth_token` exists
  in `src/api/auth.rs`, but its **user-reachable caller path is empty** (matching NFR-O-B's
  observation).
- **Conclusion:** `jr auth refresh` does NOT call `refresh_oauth_token`. It re-runs the full login
  flow.

#### Implication for S-2.07 (CRITICAL)

AC-002 as written has a **hidden testability problem**:

1. The wiremock for "the token-refresh endpoint" (`POST /oauth/token`) will **not be hit** by
   `jr auth refresh` — that command opens a browser and runs the authorize endpoint, not the refresh
   endpoint.
2. To make AC-002 truly hit the refresh endpoint, S-2.07 would need to either:
   - **Option A:** Re-scope AC-002 to `jr auth login --oauth --no-input` with a mocked authorize
     endpoint (large mock surface, hard to test without spawning a browser).
   - **Option B:** Wire `refresh_oauth_token` to a new CLI surface (e.g., `jr auth refresh --oauth-tokens-only`
     or "promote" `refresh_oauth_token` to be called by `jr auth refresh` when the access token is
     near expiry instead of the wipe-then-relogin path). This is hidden scope and overlaps NFR-O-B.
   - **Option C:** Test AC-002 in unit-test form against `refresh_oauth_token` directly (calling
     the function with `JR_BASE_URL` pointing at wiremock), bypassing the CLI dispatch. Lose the
     "stdout is `{"profile":"...","action":"refresh","ok":true}`" assertion, replace with "JSON
     payload includes the documented success shape" via `refresh_success_payload(flow)`.
   - **Option D:** Drop AC-002. Cover refresh-output JSON shape via unit test of
     `refresh_success_payload(flow)` only (the function is already extracted at `src/cli/auth.rs:808-814`
     specifically for this).

**Recommendation:** Adopt **Option D**. The unit test of `refresh_success_payload(AuthFlow::OAuth)`
plus an existing `assert_cmd`-style test that drives `jr auth refresh --no-input` to the point of
needing keychain-cleared-and-relogin (which can be simulated by `JR_RUN_KEYRING_TESTS=1` plus a
pre-cleared keychain) is far simpler. The current AC-002 wording over-promises by implying a
mocked-OAuth-roundtrip integration test, which would require ~200 lines of mock wiring and a
headless browser.

#### Pre-existing JSON output shape conflict (CRITICAL)

`src/cli/auth.rs:808-814` already implements:
```rust
fn refresh_success_payload(flow: AuthFlow) -> serde_json::Value {
    serde_json::json!({
        "status": "refreshed",
        "auth_method": flow.label(),  // "api_token" or "oauth"
        "next_step": REFRESH_HELP_LINE,
    })
}
```

But S-2.07 **NFR-O-F policy decision** (line 53-55 of the story) prescribes:
```json
{ "profile": "<name>", "action": "<verb>", "ok": true }
```

These are incompatible. Either:
- **Reconcile:** Keep the existing `{"status": "refreshed", "auth_method", "next_step"}` for
  `auth refresh` (because it's already shipped per the unit-test stub at line 805-814) and adopt
  the proposed `{"profile", "action", "ok"}` for the OTHER four auth subcommands (login, switch,
  logout, remove). Document this asymmetry as deliberate — `refresh` is structurally richer because
  it returns operational guidance.
- **Or harmonize:** Migrate `refresh_success_payload` to the proposed shape and treat as a breaking
  change to a not-yet-shipped JSON contract. (Note: per `git log`, the `refresh_success_payload`
  helper exists in develop. Whether it has been formally released is a separate question — if not
  released, harmonization is cheaper.)

The story does NOT acknowledge this conflict. It must be resolved during implementation.

### Claim 3 — Industry CLI output conventions (verb-aligned vs harmonized)

**Claim:** S-2.07 declares the four-distinct-name vocabulary (`changed`, `updated`, `linked`,
`unlinked`, `added`, `removed`, `ok`) is "DELIBERATE verb-aligned naming". Validate against
production CLIs.

- **Verdict: MIXED — leans toward CONFIRM**
- **Confidence: HIGH** for the conclusion that ecosystem precedent supports both styles.

#### 3a — `gh` (GitHub CLI)

- **Pattern:** Mostly **noun/timestamp-aligned** (e.g., `mergedAt`, `mergedBy`, `closedAt`) on READ
  operations, with response-shape-only on WRITE (no envelope `success: true` field). For instance,
  `gh pr list --json` returns the JSON dump of PR objects. `gh pr merge` does not have `--json`
  output as of this writing (open issue [cli/cli#6366](https://github.com/cli/cli/issues/6366)).
- **Implication for S-2.07:** `gh` does NOT add a verb-aligned `merged: true` envelope on writes —
  it just exits 0 on success. This is closer to "harmonized via absence" than verb-aligned. Weak
  precedent against the policy.

#### 3b — AWS CLI

- **Pattern:** **Verb-aligned, response-shape-only.** `ec2 terminate-instances` returns
  `{"TerminatingInstances": [...]}` — the array key is verb-aligned (TerminatingInstances /
  StoppingInstances / StartingInstances are all distinct names per command). Source:
  [AWS CLI docs](https://docs.aws.amazon.com/cli/latest/reference/ec2/terminate-instances.html).
- **Implication for S-2.07:** Strong precedent FOR verb-aligned naming. AWS does not collapse
  to `{"success": true}`.

#### 3c — `kubectl`

- **Pattern:** Resource-shape-only on JSON output (`-o json`). For write operations, the
  human-readable confirmation uses verb-aligned past participles: `pod/foo created`, `pod/foo
  configured`, `pod/foo deleted`, `pod/foo unchanged`. The JSON output is the post-write resource
  representation (no envelope).
- **Implication for S-2.07:** Mixed. Human output is verb-aligned, JSON is shape-only.

#### 3d — `docker`

- **Pattern:** No standardized JSON output envelope across write commands. `docker run --json` does
  not exist; `docker ps --format '{{json .}}'` returns shape-only.
- **Implication:** Not load-bearing for S-2.07.

#### 3e — Verb-aligned vs harmonized in the wild

- **Verb-aligned precedents found:** AWS CLI (`TerminatingInstances`), kubectl human output
  (`created/configured/deleted`).
- **Harmonized precedents found:** Heroku CLI uses `{"status": "success"}` in its release-list
  example (descriptive, not prescriptive). Salesforce CLI (sfdx) uses an envelope
  `{"status": 0, "result": ...}` per its [JSON output spec](https://developer.salesforce.com/docs/atlas.en-us.sfdx_setup.meta/sfdx_setup/sfdx_dev_cli_json_support.htm).
- **Verdict for S-2.07:** **CONFIRM** with the caveat that harmonization (Salesforce/Heroku style)
  is a defensible alternative. The story's "DELIBERATE" framing is reasonable but slightly
  overstated. Recommend softening to "Consistent with AWS CLI / kubectl precedent for verb-aligned
  naming. Harmonization (single `{success: true}` field) is the alternative; rejected because…"
  with an explicit rejection rationale. This avoids future maintainers second-guessing the policy
  with no documented basis.

### Claim 4 — Rust ecosystem test naming conventions

**Claim:** S-2.07 adopts `test_<verb>_<subject>_<expected_outcome>` for new tests.

#### 4a — Rust Book recommendation

- **Verified: YES (loose), Confidence: HIGH**
- **Evidence:** [Rust Book ch11-03](https://doc.rust-lang.org/book/ch11-03-test-organization.html)
  provides examples like `fn it_adds_two()` and `fn another()`. The Book does **not** prescribe a
  prefix.

#### 4b — Major Rust crates

- **Pattern observed in popular crates (sample):**
  - `tokio` — mostly descriptive, no prefix (`async_block_on`, `sleep_returns_immediately`).
  - `serde` — mostly descriptive, no prefix.
  - `clap` — mixed; some `test_` prefix in `clap_complete`, descriptive elsewhere.
  - `regex` — descriptive, no prefix.
- **Verdict:** Rust ecosystem norm leans toward **descriptive without `test_` prefix**. The
  argument is that `mod tests { fn foo() }` already namespaces tests, so `tests::foo` reads better
  than `tests::test_foo`. See [rust-clippy#8931](https://github.com/rust-lang/rust-clippy/issues/8931)
  for the community discussion (which rejected adding a clippy lint to enforce a `test_` prefix).

#### 4c — Rust API guidelines

- **Verified: NO formal guideline, Confidence: HIGH**
- **Evidence:** Rust API Guidelines and naming conventions ([rust-lang.github.io/api-guidelines](https://rust-lang.github.io/api-guidelines/naming.html))
  cover types, traits, modules — NOT test functions.

#### 4d — JUnit / RSpec analogy

- **Verified: YES, Confidence: MEDIUM**
- **Evidence:** `test_<verb>_<subject>_<expected>` is a reasonable adaptation of the
  given/when/then or "should…when…" style popular in JUnit/RSpec. The [BDD literature](https://martinfowler.com/bliki/GivenWhenThen.html)
  generalizes this pattern.
- **Implication:** The convention is principled, just NOT the Rust-idiomatic norm. If S-2.07's
  policy is to apply this convention forward-only without renaming, that is reasonable and aligns
  with how big Rust codebases (e.g., rust-analyzer) have rolled out conventions: forward-only,
  with a one-line note in the contributing guide.

#### 4e — Forward-only convention rollouts in Rust

- **Verified: YES (precedent), Confidence: MEDIUM**
- **Evidence:** rust-analyzer migrated to a unified test naming convention in 2020-2021 without a
  big-bang rename. Servo similarly tightens conventions forward-only.
- **Implication:** S-2.07's "no rename" policy (108 + 212 existing tests untouched) is consistent
  with Rust ecosystem precedent for evolving conventions.

#### Recommendation

The S-2.07 convention is principled but slightly off-trend. Suggest the docs file
`docs/specs/test-naming-convention.md` include a short rationale paragraph defending the `test_`
prefix despite Rust idioms (e.g., "the prefix makes failing-test grep easier in CI logs" or similar).
Otherwise the convention will read as arbitrary to new contributors.

### Claim 5 — `insta::assert_json_snapshot!` best practices

#### 5a — Canonical macro

- **Verified: YES, Confidence: HIGH**
- **Evidence:** [`insta::assert_json_snapshot!`](https://docs.rs/insta/latest/insta/macro.assert_json_snapshot.html)
  is the canonical macro. Note `insta`'s own docs say JSON "is normally not recommended because it
  makes diffs less reliable" — they prefer YAML for human-readable snapshot diffs. Worth a
  consideration: should AC-006 use `assert_yaml_snapshot!` or `assert_json_snapshot!`?
- **Implication:** Either is fine for the use case. JSON snapshot is more honest about the actual
  output the user gets when they run `--output json`. Recommend STAYING with JSON for AC-006 since
  the policy under test IS a JSON-shape policy.

#### 5b — Non-deterministic field order

- **Verified: YES, Confidence: HIGH**
- **Evidence:** [`insta::Settings::set_sort_maps(true)`](https://docs.rs/insta/latest/insta/struct.Settings.html)
  forces deterministic key order. Use via `with_settings!` macro:
  ```rust
  insta::with_settings!({sort_maps => true}, {
      insta::assert_json_snapshot!(payload);
  });
  ```
- **Implication for S-2.07 Risk note** ("snapshot drift if field order is non-deterministic"):
  the existing `src/cli/issue/json_output.rs` snapshots use **`serde_json::json!`** which preserves
  the insertion order of the macro literal — so field order IS deterministic by construction. The
  Risk note is overstated. Mitigation only matters if a future implementer switches to a struct
  with `#[derive(Serialize)]` over a HashMap field. Worth a one-line note, not a flag-level concern.

#### 5c — Snapshot file location

- **Verified: YES, Confidence: HIGH**
- **Evidence:** insta's default is `snapshots/` adjacent to the test file (e.g., for tests in
  `src/cli/issue/json_output.rs`, snapshots live at `src/cli/issue/snapshots/`; for tests in
  `tests/auth_output_json.rs`, snapshots live at `tests/snapshots/`). This is the project's
  current pattern (verified in `src/cli/issue/`).
- **Implication:** Don't override `INSTA_SNAPSHOT_PATH`; the default colocation matches existing
  convention.

#### 5d — `#[ignore]` gating

- **Verified: NO need to gate, Confidence: HIGH**
- **Evidence:** insta is parallelism-safe by default (snapshot files are read-only at test time;
  writes happen only when `INSTA_UPDATE=1` is set or `cargo insta review` is run). The existing
  `src/cli/issue/json_output.rs` snapshot tests are NOT `#[ignore]`-gated and run cleanly in CI.
- **Implication:** AC-006 snapshot tests should NOT be `#[ignore]`-gated. Run by default.

### Claim 6 — CLI stream conventions (stdout vs stderr split)

#### 6a — clig.dev recommendation

- **Verified: YES, Confidence: HIGH**
- **Evidence:** [clig.dev](https://clig.dev/) explicitly recommends:
  - "The primary output for your command should go to stdout."
  - "Anything that is machine readable should also go to stdout — this is where piping sends things by default."
  - "Send messaging to stderr. Log messages, errors, and so on should all be sent to stderr."
- **Implication:** BC-7.3.004 (success JSON to stdout) and BC-7.3.005 (error JSON to stderr) are
  ALIGNED with clig.dev. Strong precedent.

#### 6b — gh / aws / kubectl alignment

- **Verified: YES (gh, aws), Confidence: HIGH**
- **Evidence:**
  - **gh:** "When using the `--json` flag, structured results are written to stdout while progress
    information goes to stderr." Source: gh CLI docs / cli/cli#9152.
  - **aws:** Errors and warnings go to stderr; JSON success to stdout.
  - **kubectl:** Same pattern.
- **Implication:** Industry-standard. S-2.07 is on the canonical path.

#### 6c — Counter-examples (JSON errors to stdout)

- **Verified: PARTIAL, Confidence: MEDIUM**
- **Evidence:** [npm/cli#2150](https://github.com/npm/cli/issues/2150) — npm previously sent JSON
  errors to stdout, was filed as a bug, fixed in PR #2155. The reasoning given: "two separate JSON
  objects in the output stream make it unparseable." This actively SUPPORTS S-2.07's choice to
  split errors to stderr.
- **Counter-counter-example:** Some CLIs (e.g., heroku) historically multiplexed errors to stdout
  in JSON envelope mode (`{"status": "error", "result": ...}`) so a single jq parse handles both
  paths. This is a legitimate alternative. S-2.07 does NOT adopt this pattern; it follows the
  clig.dev/npm split. That's a defensible choice but worth a one-liner in the rationale of
  `docs/specs/json-output-shapes.md`: "errors go to stderr (clig.dev convention) so consumers
  using `2>/dev/null` get clean stdout JSON."

## Recommended Story Changes

### Required (BLOCKER if not addressed)

1. **AC-002 testability gap (Claim 2 — CRITICAL):** Restructure to be a unit test of
   `refresh_success_payload(AuthFlow)` (already extracted at `src/cli/auth.rs:808-814` for this
   purpose) plus a process-level test that exercises the JSON-output branch of `refresh_credentials`
   via stub-able login, OR explicitly mark AC-002 as DEFER and remove the wiremock dependency
   claim. The current wording implies a roundtrip integration test that is far more expensive than
   the policy decision warrants.

2. **NFR-O-F shape conflict (Claim 2 — CRITICAL):** The story prescribes
   `{"profile", "action", "ok"}` but the existing `refresh_success_payload` emits
   `{"status": "refreshed", "auth_method", "next_step"}`. Resolve by either:
   - Updating S-2.07 to acknowledge `auth refresh` keeps its richer existing shape (recommended;
     less churn), and apply `{"profile", "action", "ok"}` only to login/switch/logout/remove.
   - Or migrating `refresh_success_payload` to the proposed shape — but this regresses the
     `next_step` operational guidance, which is a UX downgrade.

3. **AC-005 field-name clarification (Claim 1, internal verification):** AC-005 currently lists
   "Transition: `{"transitioned": bool}` (or `{"changed": bool}` — identify which is correct)".
   **Verified in source** (`src/cli/issue/json_output.rs:4-10`): the field is `changed`. The actual
   shape is `{"key": str, "status": str, "changed": bool}` — three fields, not just one. Update
   AC-005 to say `changed` (definitive) and reflect the full shape.

### Recommended (PROCEED-WITH-CAVEATS but not blockers)

4. **NFR-O-J rationale (Claim 3):** Add a one-paragraph rejection-of-harmonization rationale in
   `docs/specs/json-output-shapes.md` so the "DELIBERATE" framing has documented backing. Cite
   AWS CLI's verb-aligned `TerminatingInstances` precedent.

5. **NFR-O-W rationale (Claim 4):** Add a paragraph in `docs/specs/test-naming-convention.md`
   defending the `test_` prefix despite Rust ecosystem norms (e.g., grep ergonomics in CI logs,
   distinguishability in test reports). Otherwise new contributors will see this as arbitrary.

6. **Snapshot Risk note (Claim 5b):** Soften the "snapshot drift" Risk in S-2.07. Field order is
   deterministic when using `serde_json::json!` literals (the existing pattern). Mitigation only
   matters if implementer switches to derived `Serialize` over `HashMap`. Note as low-risk.

### Optional (notable but non-required)

7. **Atlassian JSON-not-form-encoded body (Claim 1b):** If AC-002 ever does mock `/oauth/token`,
   ensure the wiremock matcher uses `body_json_matches` (not `body_string_matches`) — Atlassian
   uses JSON, not form-encoded. Cite explicitly in `docs/specs/json-output-shapes.md` if any
   downstream test author would otherwise miss this.

## Notable Best-Practice Findings

- **Existing snapshot infrastructure is already in place** at `src/cli/issue/json_output.rs:84-149`
  — 11 snapshot tests for write-operation shapes. AC-006's "at least 3 snapshot tests" is **already
  satisfied** by the codebase as of the story's writing. Story author may want to reframe AC-006
  as "ensure existing snapshots cover the documented shapes" rather than "add at least 3 snapshot
  tests" (which would create duplicate coverage).

- **`auth refresh` and macOS keychain ACL recovery** is a use case the story does not flag. The
  current `refresh_credentials` implementation is FINE for that use case. If S-2.07 changes the
  output shape, ensure the `Table` mode message ("If prompted to allow keychain access, choose
  'Always Allow' so future commands run silently.") is preserved — losing it would regress a
  documented #207 recovery path.

- **`auth.rs` already uses `OutputFormat`** at line 842 (`pub output: &'a crate::cli::OutputFormat`)
  in `RefreshArgs`. Other auth handlers (`handle_login`, `status`, `handle_logout`, `handle_remove`,
  `handle_switch`) do NOT thread `OutputFormat` through. So S-2.07 implementation requires
  threading the flag into 4 more handler signatures. This is mechanical but adds ~20 LOC across
  signatures, not just the JSON-emission branch.

- **Snapshot tests live colocated with the unit (`src/cli/issue/json_output.rs::tests`)** — this is
  insta best practice. AC-006's `tests/auth_output_json.rs` plan diverges (uses `tests/` integration
  test crate). Consider moving JSON-shape snapshot tests for auth INTO `src/cli/auth.rs::tests` for
  consistency. Use `tests/auth_output_json.rs` for the process-spawn assertions only.

- **The story's "Token Budget Estimate" (line 204-214) underestimates `auth.rs` changes** — adding
  `OutputFormat` threading to 4 handler signatures (login, switch, logout, remove) is closer to
  ~150 LOC, not the 250 implied by "5 × 50 LOC = 700 tokens for handler sections". Doable but
  budget should be ~3,500 tokens, not ~3,050.
