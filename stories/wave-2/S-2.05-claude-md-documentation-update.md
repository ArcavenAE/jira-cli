**AC-004** (traces to BC-6.1.001 postcondition, NFR-O-V): CLAUDE.md conventions or

**AC-006** (traces to BC-6.1.001 postcondition, NFR-O-R): CLAUDE.md conventions or
Architecture section documents the five output-channel profiles used by CLI handlers:
1. **Pure**: only stdout (JSON or table data), no stderr
2. **Read-only**: stdout for data, stderr for hints/warnings
3. **Mixed**: stdout for success data, stderr for errors + hints
4. **Symmetric**: stdout for `--output json`, stderr for errors in either mode
5. **No-log facade**: state-changing commands that emit only JSON on stdout

### NFR-O-M: --open dual mechanism

Add to the Gotchas section (after existing entries):

```markdown
- **`--open` filter uses two mechanisms:** `statusCategory != Done` is injected into
  the JQL query for Jira issues (server-side filter). For connected tickets in
  `cli/assets.rs:303-321`, filtering is client-side via `status.colorName != "green"`.
  Both mechanisms implement the same user intent ("show only open items") but at
  different layers. This is intentional — CMDB connected tickets do not support JQL
  `statusCategory` filtering. Status category colors are hardcoded: `green` = Done,
  `yellow` = In Progress, `blue-gray` = To Do.
```

### NFR-O-O: user pagination JRACLOUD-71293 workaround

Add to `src/api/jira/users.rs` near the pagination loop:
```rust
// JRACLOUD-71293: Jira Cloud may return a short non-empty page that does NOT
// indicate the end of results. We advance by USER_PAGE_SIZE (not by returned count)
// to avoid re-fetching the same records. A short page is normal; only an EMPTY page
// signals end of results.
```

Add to CLAUDE.md Gotchas:
```markdown
- **User pagination advances by `USER_PAGE_SIZE` not returned-count:**
  `src/api/jira/users.rs` increments `start_at` by `USER_PAGE_SIZE` (100) after each
  page, NOT by the number of users returned. This workaround for JRACLOUD-71293
  prevents re-fetching on short non-empty pages. A future "fix" that switches to
  returned-count would introduce duplicates. Do not change this behavior.
```

### NFR-R-F: search_issues cursor loop

In `src/api/jira/issues.rs` near the `search_issues` cursor loop:
```rust
// KNOWN-GAP: NFR-R-F. get_changelog (lines 222-230) has an anti-loop guard that
// breaks when nextPage URL == current URL. search_issues currently has no analogous
// guard against cursor == cursor regression. In practice, Jira does not send
// identical cursors; this gap is LOW risk. If observed, add:
//   if next_cursor == cursor { break; }
// See nfr-catalog.md NFR-R-F for context.
```

### NFR-O-V and NFR-O-R

Add concise entries to CLAUDE.md. NFR-O-V goes under Gotchas. NFR-O-R can go under
Conventions as a "Output channels" subsection.
