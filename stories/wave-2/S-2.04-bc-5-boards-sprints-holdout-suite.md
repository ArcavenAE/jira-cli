# S-2.04: BC-5/7 boards, sprints, and output rendering holdout suite (H-040..H-044)

## Context

The boards and sprints bounded context (`src/cli/sprint.rs`, `src/cli/board.rs`,
`src/cli/issue/format.rs`) implements several display and behavioral contracts
that are easy to regress during Phase 3 changes. The ADF rendering bounded context
(`src/adf.rs`) is also at risk from any future node-type additions.

This story pins five holdout scenarios:

- **H-040**: `sprint current` truncation — 30-default, `--all` bypasses, under-limit
  produces no hint. Three-case contract.
- **H-041**: `sprint add` vs `sprint remove` JSON shape asymmetry — `sprint add`
  includes `sprint_id`; `sprint remove` does NOT. Pin against "harmonization".
- **H-042**: `sprint list --board N` on a kanban board is a hard error with exact
  literal message `Sprint commands are only available for scrum boards`.
- **H-043**: Team column in `sprint current` output — appears only when BOTH
  `team_field_id` is configured AND at least one issue has a team UUID (conjunctive gate).
- **H-044**: `issue view` with ADF description — renders text nodes (heading, paragraph,
  code block), silently drops mention nodes, no panic on any ADF node type.

All five are MUST-PASS from activation HEAD dea1664.

## Behavioral Contracts

**BC-5.2.005**: `sprint current` shows at most 30 issues by default. When the sprint
has >30 issues, stderr contains `"Showing 30 results"` hint. When `--all` is passed,
all issues are shown (no hint). When ≤30 issues are present, no hint is shown.

**BC-5.2.007**: `sprint add --sprint N KEY...` returns JSON `{"added": true, "issues": [...], "sprint_id": N}`.
The `sprint_id` field IS present in the add response.

**BC-5.2.008**: `sprint remove --sprint N KEY...` returns JSON `{"issues": [...], "removed": true}`.
The `sprint_id` field is NOT present in the remove response.

**BC-5.2.001**: `sprint list --board N` (and other sprint subcommands) on a kanban board
exits non-zero with the exact message `Sprint commands are only available for scrum boards`.

**BC-5.3.001**: The team column in `sprint current` output appears only when
`team_field_id` is configured in the active profile AND at least one issue has a non-null
team UUID.

**BC-5.3.002**: The team column is absent when all issues have null team UUID even if
`team_field_id` is configured.

**BC-7.2.001**: `issue view` on an issue with ADF description renders the plain-text
content of heading, paragraph, and code-block nodes. Mention nodes are silently dropped
(current behavior per issue #202). No panic occurs on any ADF node type (including
unknown future node types that fall through the match).

## Acceptance Criteria

**AC-001** (traces to BC-5.2.005 postcondition): Given three wiremock sprint setups:
(a) sprint with 35 issues, (b) sprint with 35 issues + `--all` flag, (c) sprint with
10 issues, when `jr sprint current` (or `jr sprint current --all` for case b) is
invoked, then:
- Case (a): exactly 30 results in output, stderr contains `"Showing 30 results"` or `"~"`.
- Case (b): exactly 35 results in output, no truncation hint in stderr.
- Case (c): exactly 10 results in output, no truncation hint in stderr.
(Pins H-040, three-case contract.)

**AC-002** (traces to BC-5.2.007 postcondition): Given a sprint ID of 100 and issues
`["TEST-1", "TEST-2"]`, when `jr sprint add --sprint 100 TEST-1 TEST-2 --output json`
is invoked, then stdout JSON equals `{"added": true, "issues": ["TEST-1", "TEST-2"], "sprint_id": 100}`.
The `sprint_id` key IS present. (Pins H-041 add path.)

**AC-003** (traces to BC-5.2.008 postcondition): Given the same sprint and issues,
when `jr sprint remove --sprint 100 TEST-1 TEST-2 --output json` is invoked, then
stdout JSON equals `{"issues": ["TEST-1", "TEST-2"], "removed": true}`.
The `sprint_id` key is NOT present. (Pins H-041 remove path.)

**AC-004** (traces to BC-5.2.001 postcondition): Given a wiremock board endpoint
returning `type = "kanban"`, when `jr sprint list --board 1` is invoked, then exit
code is non-zero and stderr contains the exact literal string
`Sprint commands are only available for scrum boards`. (Pins H-042.)

**AC-005** (traces to BC-5.3.001 postcondition): Given `team_field_id` configured
and an issue list where one issue has a team UUID (resolved from team cache), when
`jr sprint current` is invoked, then the output table includes a `Team` column.
(Pins H-043 case a.)

**AC-006** (traces to BC-5.3.002 postcondition): Given `team_field_id` configured
but NO issue in the sprint has a team UUID, when `jr sprint current` is invoked, then
the output table does NOT include a `Team` column. (Pins H-043 case b.)

**AC-007** (traces to BC-7.2.001 postcondition): Given an issue `PROJ-1` with an ADF
description containing a `heading` node, a `paragraph` node, a `codeBlock` node, and
a `mention` node, when `jr issue view PROJ-1` is invoked, then:
- Exit code is 0.
- stdout contains the text of the heading node.
- stdout contains the text of the paragraph node.
- The mention node text is NOT in stdout (silently dropped per current behavior).
- No panic occurs.
(Pins H-044.)

## Implementation Notes

### H-040 (sprint current truncation — three cases)

```rust
// Case (a): 35 issues → expect 30 + hint
// Case (b): 35 issues + --all → expect 35, no hint
// Case (c): 10 issues → expect 10, no hint
```

Note: "30 results" is the truncation threshold. Verify the exact constant in
`src/cli/sprint.rs`. The stderr hint pattern (`Showing 30 results` or `~`) must
match exactly what the code emits.

### H-041 (add vs remove JSON asymmetry)

Parse stdout JSON for both commands. For the add command, confirm `sprint_id` key
exists. For the remove command, confirm `sprint_id` key is ABSENT. Use serde_json
for parsing.

### H-042 (kanban hard error)

The board type check happens when fetching board config. Wiremock board config endpoint:
- `GET /rest/agile/1.0/board/1/configuration` → body with `type: "kanban"`

Assert stderr contains the EXACT literal string. A future refactor that changes the
wording would be caught.

### H-043 (team column conjunctive gate)

The team column requires:
1. `team_field_id` in config (set in the `.jr.toml` temp file or profile config).
2. At least one issue with a non-null team UUID in the `team_field_id` custom field.
3. The team UUID must resolve to a name from the teams cache.

For case (a): write a `teams.json` to `XDG_CACHE_HOME/jr/v1/default/teams.json`
containing `{"teams": [{"id": "<uuid>", "displayName": "Engineering"}]}`.
Configure the issue mock to return a team UUID matching this entry.

For case (b): same setup but ALL issues have `null` for the team field.

### H-044 (ADF rendering)

Construct an ADF description JSON fixture with:
- A `heading` node (level 1, "My Heading")
- A `paragraph` node ("Some text")
- A `codeBlock` node
- A `mention` node with `attrs.id` and `attrs.text`

Verify `jr issue view PROJ-1` stdout contains "My Heading" and "Some text".
Verify the mention text is NOT in stdout.
Verify no panic (exit code 0).

Note per NFR-O-I: mention nodes are INTENTIONALLY silently dropped in current code
(`adf.rs:531-540`). H-044 pins this as the current (correct) behavior. When NFR-O-I
is fixed (Wave 3+), H-044's mention assertion must flip.
