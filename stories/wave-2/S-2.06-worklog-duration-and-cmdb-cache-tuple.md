# S-2.06: Worklog timeSpent server-side parsing (NFR-R-C) and CMDB cache tuple format pin

## Context

This is a v2.0.0 rewrite of S-2.06. The v1.0.0 design proposed fetching Jira's
`/rest/api/3/configuration/timetracking` endpoint to obtain `hoursPerDay`/`daysPerWeek`
for client-side duration math. A Perplexity verification pass on 2026-05-08 found three
blocking factual errors in that approach:

1. **Wrong endpoint:** `GET /rest/api/3/configuration/timetracking` returns the active
   time-tracking provider (name/key), not working-hours config. The actual working-hours
   endpoint is `/rest/api/3/configuration/timetracking/options`.
2. **Wrong field names and types:** The documented fields are `workingHoursPerDay` and
   `workingDaysPerWeek` (floats, e.g. 7.6 / 5.5), not `hoursPerDay`/`daysPerWeek`
   (integers).
3. **Admin-only:** Both endpoints require "Administer Jira" global permission. Non-admin
   users receive 403 regardless of OAuth scope. The `jr` typical user is not an admin.

The chosen resolution (Option 1) is the same pattern used by `ankitpokhrel/jira-cli`
(the canonical Go Jira CLI): forward the user's duration string directly to the worklog
POST as the `timeSpent` field (a string). Jira parses it server-side using the instance's
own configured `workingHoursPerDay`/`workingDaysPerWeek`. No client-side math, no API
fetch, no admin-permission dependency.

See `.factory/research/S-2.06-jira-timetracking-verification.md` for the full
verification report.

This story addresses two concerns:

### Part A: NFR-R-C — Worklog timeSpent string passthrough

`src/cli/worklog.rs` currently calls `duration::parse_duration(dur, 8, 5)` and sends the
result as `timeSpentSeconds` (an integer). This silently produces wrong values on any
Jira instance with non-standard working hours (e.g., 7.5h/day or 4-day weeks).

**Fix:** Change the worklog POST to use `timeSpent` (string) instead of `timeSpentSeconds`.
Downgrade `parse_duration` from a calculator to a syntactic validator that confirms the
input matches Jira's accepted format (`Nw Nd Nh Nm`) and rejects garbage before any
network call. Jira applies its own configured hours-per-day and days-per-week when it
processes the `timeSpent` string server-side.

### Part B: CMDB cache tuple format pin

Per CLAUDE.md Gotchas: "`~/.cache/jr/v1/<profile>/cmdb_fields.json` stores `(id, name)`
tuples. Old format (ID-only) causes deserialization failure, handled as cache miss."

A regression test is needed to pin this graceful-degradation invariant and prevent a
future "simplification" from introducing a panic.

## Behavioral Contracts

**BC-X.5.009** (worklog duration): `parse_duration` is a syntactic validator. It
confirms the string matches Jira's accepted duration format (`Nw Nd Nh Nm` with valid
positive digits and at least one unit). It does NOT compute seconds, and it does NOT
accept `hours_per_day` or `days_per_week` parameters. When the input is valid, the
user's original string is forwarded as `timeSpent` in the worklog POST body. When the
input is invalid, the command exits with code 64 and a helpful syntax message; no POST
is made.

**BC-6.2.006** (cache resilience): When `cmdb_fields.json` contains a format that
fails deserialization (e.g., ID-only `Vec<String>` from an older version), the cache
read returns a miss (None) instead of panicking or returning an error. The caller
proceeds to re-fetch from the API and overwrites the cache with the new tuple format.

## Acceptance Criteria

### Part A: NFR-R-C — timeSpent string passthrough

**AC-001** (traces to BC-X.5.009 postcondition — POST body): Given a wiremock server
serving the worklog POST endpoint, when `jr worklog add PROJ-1 --time "1d"` is invoked,
then the captured POST body contains `"timeSpent": "1d"` as a string field, and
`timeSpentSeconds` is NOT present in the body. The server applies its own
`workingHoursPerDay` when interpreting the duration.

**AC-002** (traces to BC-X.5.009 postcondition — mixed units preserved): Given a
wiremock server serving the worklog POST endpoint, when `jr worklog add PROJ-1
--time "2d 3h 30m"` is invoked, then the captured POST body contains
`"timeSpent": "2d 3h 30m"` with spaces and mixed units preserved exactly as entered.
`timeSpentSeconds` is NOT present in the body.

**AC-003** (traces to BC-X.5.009 precondition — validation rejects garbage): Given an
invalid duration string `"1z"`, when `jr worklog add PROJ-1 --time "1z"` is invoked,
then (a) exit code is 64, (b) stderr contains a syntax hint mentioning `Nw Nd Nh Nm`,
(c) zero POST requests are made to the worklog endpoint (use `wiremock::expect(0)` or
equivalent). The validator catches the error before any network call.

**AC-004** (traces to BC-X.5.009 unit — validator contract): The refactored
`parse_duration_validate("1d")` (or equivalent renamed function) returns `Ok`. The call
`parse_duration_validate("1z")` returns `Err`. The function has NO `hours_per_day` or
`days_per_week` parameters. (Unit test in `src/duration.rs` or
`tests/worklog_duration_holdouts.rs` verifying the new validator contract.)

### Part B: CMDB cache tuple format

**AC-005** (traces to BC-6.2.006 postcondition — graceful degradation): Given
`<XDG_CACHE_HOME>/jr/v1/default/cmdb_fields.json` containing `["customfield_10191"]`
as a `Vec<String>` (legacy ID-only format) and a wiremock at `JR_BASE_URL` returning
the canonical CMDB-field list as a JSON array of `{id, name}` objects, when
`jr issue list --output json --project PROJ` is invoked, then (a) exit code is 0,
(b) no panic or hard error occurs, (c) the cache file is overwritten with the new
`Vec<(String, String)>` tuple format.

**AC-006** (traces to BC-6.2.006 invariant — cache hit on valid format): When
`cmdb_fields.json` contains the current `(id, name)` tuple format, the cache read
succeeds and returns the correct data. No CMDB-fields API call is made when a valid
cache entry exists. (Verify with `wiremock::expect(0)` on the fields endpoint.)

## Implementation Notes

### Part A: timeSpent string passthrough

**`src/api/jira/worklogs.rs` — change the POST body:**

```rust
// Before:
pub async fn add_worklog(
    &self,
    key: &str,
    time_spent_seconds: u64,
    comment: Option<Value>,
) -> Result<Worklog> {
    let mut body = serde_json::json!({
        "timeSpentSeconds": time_spent_seconds,
    });
    ...
}

// After:
pub async fn add_worklog(
    &self,
    key: &str,
    time_spent: &str,
    comment: Option<Value>,
) -> Result<Worklog> {
    let mut body = serde_json::json!({
        "timeSpent": time_spent,
    });
    ...
}
```

The `timeSpentSeconds` field is removed from the POST body entirely. The Jira API
accepts either `timeSpent` (string) or `timeSpentSeconds` (integer); they are mutually
exclusive when the server needs to apply working-hours conversion.

**`src/duration.rs` — refactor parse_duration to a validator:**

The current `parse_duration(input, hours_per_day, days_per_week) -> Result<u64>` must
be refactored. The simplest approach that preserves all existing callers:

1. Rename the current function to `parse_duration_validate(input: &str) -> Result<()>`
   (or `-> Result<&str>` if a normalized form is desired).
2. Remove the `hours_per_day` and `days_per_week` parameters.
3. Replace all arithmetic (`num * days_per_week * hours_per_day * 3600`, etc.) with
   unit recognition only — the function verifies the unit character is valid (`w`, `d`,
   `h`, `m`) but does not accumulate seconds.
4. Return `Ok(())` (or `Ok(input.trim())`) on valid input; `Err` on invalid.
5. Delete `format_duration` usage from `handle_add` in `worklog.rs` since we no longer
   have a seconds value to format. The success message can use the original string:
   `output::print_success(&format!("Logged {} on {}", time_str, key))`.
6. Confirm via `rg "parse_duration" --type rust` that no other callers remain before
   deleting the old signature. If proptests reference the old signature, update them
   to use the new validator.

**`src/cli/worklog.rs` — wire string passthrough:**

```rust
// Before:
let seconds = duration::parse_duration(dur, 8, 5)?;
let comment = message.map(adf::text_to_adf);
let worklog = client.add_worklog(key, seconds, comment).await?;
...
let formatted = duration::format_duration(seconds);
output::print_success(&format!("Logged {formatted} on {key}"));

// After:
duration::parse_duration_validate(dur)?;
let comment = message.map(adf::text_to_adf);
let worklog = client.add_worklog(key, dur, comment).await?;
...
output::print_success(&format!("Logged {} on {}", dur, key));
```

The user's original string is forwarded to Jira verbatim. No seconds variable needed
in `handle_add`. The `format_duration` call in `handle_add` is removed (it was only
needed to pretty-print the computed seconds; the original string is already human-readable).

Note: `format_duration` is still used in `handle_list` for displaying existing worklogs
that return `time_spent_seconds` from the API. Do NOT remove `format_duration` — only
stop calling it from `handle_add`.

### Part B: CMDB cache regression test

Write a test that:
1. Writes `["customfield_10191"]` (ID-only, legacy `Vec<String>` format) to the CMDB
   fields cache path.
2. Calls the cache read function for CMDB fields.
3. Asserts the result is `None` (cache miss on format failure).
4. Asserts no panic.

This test can be inline in `tests/worklog_duration_holdouts.rs` if the cache function
is accessible from integration tests, or in `src/cache.rs` as a `#[cfg(test)]` unit
test if the function is private. Prefer inline `#[cfg(test)]` in `cache.rs` for
private-function access.

## Test Plan

**Red Gate (Part A):** AC-001 fails before implementation — the current code sends
`timeSpentSeconds` (an integer) in the POST body, but the test expects `timeSpent`
(a string) and asserts `timeSpentSeconds` is absent. The existing H-018 unit tests
for `parse_duration` with `hours_per_day`/`days_per_week` parameters will also fail
once the function signature is changed (forcing those to be updated as part of this story).

**Red Gate (Part B):** Write the ID-only format test first. If the cache read already
returns `None` on format failure, the test passes immediately (the invariant was already
correct). If it panics instead, that is the bug to fix. Either way, the test pins the
behavior going forward.

**Implementation order:**
1. Part B: Write CMDB cache regression test. Verify graceful degradation passes or fix.
2. Part A: Refactor `parse_duration` to `parse_duration_validate` (no calculator, no
   `hours_per_day`/`days_per_week` params). Update all call sites and existing unit tests.
3. Part A: Change `add_worklog` signature from `time_spent_seconds: u64` to
   `time_spent: &str`; update POST body to `"timeSpent"`.
4. Part A: Wire `worklog.rs::handle_add` to use validator + string passthrough.
5. Part A: Write AC-001, AC-002, AC-003 integration tests (wiremock body capture).
6. Part A: Write AC-004 unit test for new validator contract.
7. Run full test suite: `cargo test && cargo clippy -- -D warnings`.

**Proptest update:** `src/duration.rs::proptests` currently calls
`parse_duration(&input, 8, 5)`. After the refactor, update these to call
`parse_duration_validate(&input)` and verify `is_ok()` for valid inputs.

## Architecture Compliance Rules

- **No new files in `src/api/jira/`.** The planned `timetracking.rs` is permanently
  cancelled. Do NOT create it.
- **No new cache entries.** The planned timetracking config cache entry is permanently
  cancelled. Do NOT add it to `src/cache.rs`.
- **`add_worklog` signature change is breaking at the call site** (`cli/worklog.rs`).
  Update both the definition and the call site in the same commit to avoid a broken
  intermediate state.
- **`parse_duration` refactor:** if the old function name is kept, change its signature
  by removing `hours_per_day` and `days_per_week`. If renamed to `parse_duration_validate`,
  ensure all callers (including proptests) are updated. Do not leave a dead-code function
  with `#[allow(dead_code)]`.
- **Multi-profile cache boundary (Part B):** any cache read/write function must take
  `profile: &str` as its first argument. This is unchanged from v1.0.0.
- **`format_duration` is still needed** for `handle_list` (displaying existing worklogs
  that return `time_spent_seconds` from the API). Do not remove it.
- Reference: `architecture/module-decomposition.md` for `api/jira/` and `cache.rs`
  patterns.

## Risk / Dependencies

- **No Wave 0 or Wave 1 dependencies.** Independent of all other Wave 2 stories.
- **Simplification vs v1.0.0.** This design is strictly simpler: fewer files, no new
  cache entry, no admin-permission concern, no fallback logic. Risk of regression is
  lower than v1.0.0.
- **`parse_duration` callers.** Run `rg "parse_duration" --type rust` before and after
  the refactor to confirm all callers are updated. The proptests in `src/duration.rs`
  reference the old signature and must be updated.
- **`format_duration` still needed.** `handle_list` uses it for display. Confirm it is
  not removed by mistake.
- **No breaking change for end users.** The `--time` flag syntax is unchanged. Users
  still type `jr worklog add PROJ-1 --time "1d 2h"`. The difference is transparent:
  the server now interprets the duration using its own configuration rather than the
  hardcoded 8h/day, 5d/week constants.
