# S-2.06: Worklog duration config fetch (NFR-R-C) and CMDB cache tuple format pin

## Context

This story addresses two related reliability concerns in the cache and duration
handling code:

### Part A: NFR-R-C — Worklog duration hardcoded constants

`src/cli/worklog.rs:32` passes hardcoded constants `hours_per_day = 8` and
`days_per_week = 5` to `parse_duration`. Jira Cloud instances can configure these
values via `/rest/api/3/configuration/timetracking`. A 7.5h/day or 4-day week
Jira instance silently receives wrong worklog durations.

**Fix**: Fetch timetracking config from Jira API at `GET /rest/api/3/configuration/timetracking`.
Cache the result for 7 days (same TTL as other caches). Fall back to 8/5 on cache miss
or API error.

### Part B: CMDB cache tuple format pin

Per CLAUDE.md Gotchas: "`~/.cache/jr/v1/<profile>/cmdb_fields.json` stores `(id, name)`
tuples. Old format (ID-only) causes deserialization failure, handled as cache miss."

A regression test that writes an ID-only CMDB fields cache and verifies graceful
degradation (cache-miss behavior, NOT panic or hard error) is needed to pin this
invariant. This prevents a future "simplification" of the deserialization that silently
introduces a crash.

## Behavioral Contracts

**BC-X.9.001** (worklog duration): `parse_duration` receives `hours_per_day` and
`days_per_week` from the Jira timetracking configuration, not hardcoded constants.
When the API returns custom values, those values are used. When the cache contains
valid config, no API call is made. When the API is unavailable or returns an error,
fall back to `(8, 5)` with no user-visible error.

**BC-6.2.013** (cache resilience): When `cmdb_fields.json` contains a format that
fails deserialization (e.g., ID-only format from an older version), the cache read
returns a miss (None) instead of panicking or returning an error. The caller proceeds
to re-fetch from the API.

## Acceptance Criteria

### Part A: NFR-R-C

**AC-001** (traces to BC-X.9.001 postcondition): Given a Jira instance that returns
`{"hoursPerDay": 7, "daysPerWeek": 4}` from `GET /rest/api/3/configuration/timetracking`,
when `jr worklog add PROJ-1 --time "1d"` is invoked, then the worklog POST body
contains `timeSpentSeconds = 7 * 3600 = 25200` (NOT `8 * 3600 = 28800`).

**AC-002** (traces to BC-X.9.001 postcondition): Given a successful first call and a
cached timetracking config, when a second `jr worklog add` command is invoked, then
the timetracking endpoint receives exactly ONE call total (first hits HTTP; second uses
cache). Use wiremock `expect(1)`.

**AC-003** (traces to BC-X.9.001 postcondition): Given a Jira instance that returns
500 from the timetracking endpoint, when `jr worklog add PROJ-1 --time "1d"` is invoked,
then the worklog POST body contains `timeSpentSeconds = 8 * 3600 = 28800` (fallback
to default 8h/day). No error is surfaced to the user for the timetracking config fetch
failure.

**AC-004** (traces to BC-X.9.001 postcondition): Given a Jira instance that returns
`{"hoursPerDay": 8, "daysPerWeek": 5}` (default values), when `parse_duration("1w", 8, 5)`
is called, then the result equals `5 * 8 * 3600 = 144000` seconds. (Unit test verifying
the duration parser receives the fetched config, not a hardcoded literal.)

### Part B: CMDB cache tuple format

**AC-005** (traces to BC-6.2.013 postcondition): Given a `cmdb_fields.json` file in
`XDG_CACHE_HOME` containing an ID-only format (e.g., `["customfield_10191"]` as a
`Vec<String>` instead of `Vec<(String, String)>`), when any command that reads CMDB
fields is invoked, then the cache returns a miss (no panic, no hard error), the command
proceeds to re-fetch from the API, and exit code is 0.

**AC-006** (traces to BC-6.2.013 invariant): When `cmdb_fields.json` contains the
current `(id, name)` tuple format, the cache read succeeds and returns the correct data.
No API call is made for CMDB fields when a valid cache entry exists.

## Implementation Notes

### Part A: Timetracking API

**New file**: `src/api/jira/timetracking.rs`:
```rust
pub struct TimetrackingConfig {
    pub hours_per_day: u32,
    pub days_per_week: u32,
}

impl Default for TimetrackingConfig {
    fn default() -> Self {
        Self { hours_per_day: 8, days_per_week: 5 }
    }
}

impl JiraClient {
    pub async fn get_timetracking_config(&self) -> Result<TimetrackingConfig> {
        let config: serde_json::Value = self
            .get("/rest/api/3/configuration/timetracking")
            .await?;
        Ok(TimetrackingConfig {
            hours_per_day: config["hoursPerDay"].as_u64().unwrap_or(8) as u32,
            days_per_week: config["daysPerWeek"].as_u64().unwrap_or(5) as u32,
        })
    }
}
```

**Cache integration** in `src/cache.rs`: Add a `timetracking_config` cache entry with
7-day TTL, following the same pattern as `workspace_id` and `teams`.

**`src/cli/worklog.rs`**: Replace `hours_per_day: 8, days_per_week: 5` literals with
a call to `get_timetracking_config()` → cache → fallback.

```rust
// Before:
let seconds = parse_duration(&time_str, 8, 5)?;

// After:
let tt_config = client.get_timetracking_config_cached(&config.active_profile_name).await
    .unwrap_or_default();
let seconds = parse_duration(&time_str, tt_config.hours_per_day, tt_config.days_per_week)?;
```

### Part B: CMDB cache format regression

Write a `#[test]` that:
1. Writes `["customfield_10191"]` (ID-only, legacy format) to the CMDB fields cache path.
2. Calls the cache read function for CMDB fields.
3. Asserts the result is `None` (cache miss on format failure).
4. No panic.

This test should be in `tests/worklog_duration_holdouts.rs` or inline in `cache.rs`
depending on visibility.

## Test Plan

## Risk / Dependencies

- **No Wave 0 or Wave 1 dependencies.** Independent of other Wave 2 stories.
- **Risk: `api/jira/mod.rs` structure.** Some jira API files may be declared differently
  (direct `use` vs `mod` in `api/jira/mod.rs`). Verify the module declaration pattern
  before adding `timetracking.rs`.
- **Risk: timetracking endpoint availability.** Not all Jira instances may have the
  timetracking API enabled. The fallback to 8/5 handles this gracefully (AC-003).
- **Risk: CMDB fields cache path.** Confirm exact path in `cache.rs` before writing
  the Part B test.
- **No breaking change.** The `parse_duration` signature is unchanged; the caller
  now fetches the values instead of hardcoding them.
