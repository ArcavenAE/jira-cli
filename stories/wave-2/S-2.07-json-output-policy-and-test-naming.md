# S-2.07: Auth --output json (4 subcommands) + verb-aligned JSON policy + test naming convention

## Context

This is a v2.0.0 rewrite of S-2.07. The v1.0.0 design contained three concrete errors
discovered via Perplexity-driven source verification on 2026-05-08. The corrections are
captured in `.factory/research/S-2.07-json-policy-and-conventions-research.md`.

**What changed from v1.0.0:**

1. **AC-002 rewritten** — v1.0.0 specified an end-to-end test mocking a token-refresh
   endpoint (POST `https://auth.atlassian.com/oauth/token`). Source verification found
   that `jr auth refresh` does NOT call that endpoint: it wipes credentials and re-runs
   the full OAuth 3LO login flow via `login_oauth` (verified at `src/cli/auth.rs:895-909`).
   The wiremock premise was structurally untestable. v2.0.0 replaces it with a unit test
   of the already-extracted `refresh_success_payload(AuthFlow)` helper
   (`src/cli/auth.rs:808-814`), which is the correct regression pin.

2. **Auth JSON shape asymmetry documented** — NFR-O-F v1.0.0 prescribed a uniform
   `{"profile", "action", "ok"}` shape for all five auth subcommands including `refresh`.
   Source verification found that `refresh_success_payload` was already shipped with shape
   `{"status": "refreshed", "auth_method", "next_step"}` (verified at `src/cli/auth.rs:808-814`).
   v2.0.0 preserves the existing `refresh` shape and applies `{"profile", "action", "ok"}`
   only to the four NEWLY-JSON-emitting subcommands: `login`, `switch`, `logout`, `remove`.
   The asymmetry is intentional and documented (see Implementation Notes below).

3. **AC-005 `changed` field verified** — v1.0.0 said "Transition: `{transitioned: bool}` (or
   `{changed: bool}` — identify which is correct)". Verified in source at
   `src/cli/issue/json_output.rs:4-10`: the field is `changed`, full shape is
   `{"key": str, "status": str, "changed": bool}`. AC-005 is now definitive. This also
   **closes S-2.02-DEFER** — the BC-3.2.001 spec language was using `"transitioned"` while
   the code emits `"changed"`. v2.0.0 permanently documents `changed` as canonical.

4. **AC-006 reframed** — v1.0.0 said "add at least 3 snapshot tests". Source verification
   found that `src/cli/issue/json_output.rs:84-149` already has 11 `insta::assert_json_snapshot!`
   tests covering all write-operation shapes. The "add 3" goal was already satisfied.
   v2.0.0 reframes AC-006 as: extend the existing 11-test suite by adding 4 new snapshots
   for the auth subcommands (`login`, `switch`, `logout`, `remove`).

### NFR-O-F: Auth subcommands missing `--output json` paths

Four auth subcommands (`login`, `switch`, `logout`, `remove`) have no `--output json`
handling. They emit only human-readable output regardless of the `--output` flag. This
violates BC-7.1.001 that `--output json` emits structured JSON for all commands that support the flag.

Note: `auth refresh` already has JSON output via `refresh_success_payload` (shape documented
in Implementation Notes). This story does NOT modify the `refresh` handler.

**Policy decision**: Add JSON output for `login`, `switch`, `logout`, `remove` using:

```json
{ "profile": "<name>", "action": "<verb>", "ok": true }
```

Error shape (to stderr) per BC-7.3.005: `{"error": "<message>", "code": <N>}`.

### NFR-O-J: JSON field naming — verb-aligned, DELIBERATE

Write operations use distinct boolean success fields: `"changed"` (transitions, assign),
`"updated"` (issue edit), `"linked"` / `"unlinked"` (issue links), `"added"` / `"removed"`
(sprint). This vocabulary is DELIBERATE verb-aligned naming consistent with AWS CLI
(`TerminatingInstances`, `StartingInstances`) and kubectl human output (`created`,
`configured`, `deleted`). The policy is DOCUMENTED and snapshot-pinned, not changed.

### NFR-O-W: Mixed test prefix styles

108 tests use `test_<verb>_<subject>` prefix; 212 tests use `<subject>_<verb>_<expected>`
with no prefix. **Policy decision**: Adopt `test_<verb>_<subject>_<expected_outcome>` as
the canonical convention for all new tests. Existing tests are NOT renamed. The convention
is documented and applied to tests written in this story.

## Behavioral Contracts

**BC-7.1.001**: `--output json` emits structured JSON for all commands that support the flag.
Auth subcommands (`login`, `switch`, `logout`, `remove`) must participate. `auth refresh`
already participates via `refresh_success_payload`.

**BC-7.4.013**: `auth login --output json` emits `{"profile": <name>, "action": "login", "ok": true}` to stdout on success.

**BC-7.4.014**: `auth switch --output json` emits `{"profile": <name>, "action": "switch", "ok": true}` to stdout on success.

**BC-7.4.015**: `auth logout --output json` emits `{"profile": <name>, "action": "logout", "ok": true}` to stdout on success.

**BC-7.4.016**: `auth remove --output json` emits `{"profile": <name>, "action": "remove", "ok": true}` to stdout on success.

**BC-7.3.005**: Error outputs when `--output json` is set are structured JSON to stderr:
`{"error": "<message>", "code": <N>}` where `code` matches the exit code.

is project-local, not ecosystem-prescriptive.

## Implementation Notes

### Auth JSON branches (login/switch/logout/remove)

In `src/cli/auth.rs`, add surgical `--output json` branches to the four handler functions:
`handle_login`, `handle_switch`, `handle_logout`, `handle_remove`. Use the existing
`OutputFormat` enum pattern (same as `issue list`, `issue view`).

Note from research: `auth.rs` uses `OutputFormat` in `RefreshArgs` at line 842
(`pub output: &'a crate::cli::OutputFormat`), but the other four handlers (`handle_login`,
`handle_switch`, `handle_logout`, `handle_remove`) do NOT currently thread `OutputFormat`
through their signatures. Implementation requires threading `OutputFormat` into 4 handler
signatures — mechanical but ~20 LOC across signatures, not just the JSON-emission branches.

```rust
// Pattern: at the end of each handler, after success:
if output == &OutputFormat::Json {
    let json = serde_json::json!({
        "profile": profile_name,
        "action": "switch",  // or "login", "logout", "remove"
        "ok": true
    });
    println!("{}", json);
} else {
    eprintln!("Switched to profile '{}'", profile_name);
}
```

DO NOT touch the `refresh` handler (`refresh_credentials` / `refresh_success_payload`).
The refresh shape `{"status": "refreshed", "auth_method", "next_step"}` is preserved as-is.

### Asymmetric `auth refresh` shape rationale

`auth refresh` keeps `{"status": "refreshed", "auth_method": str, "next_step": str}` while
`login/switch/logout/remove` use `{"profile": str, "action": str, "ok": true}`. The asymmetry
is intentional: `refresh` triggers re-authentication (a flow — wipe credentials, re-run full
OAuth 3LO or api-token login), not merely a state change. Its JSON payload communicates
operational guidance (`next_step` = keychain ACL recovery hint per issue #207). A uniform
`ok: true` would discard that guidance. The four state-change commands (`login`, `switch`,
`logout`, `remove`) share a simpler profile/action/ok envelope because they have no
operational guidance to communicate.

This asymmetry MUST be documented in `docs/specs/json-output-shapes.md` (AC-005).

### AC-002 unit test placement

Place the AC-002 unit test inline in `src/cli/auth.rs::#[cfg(test)] mod tests`. An existing
test at approximately line 1290 (`test_refresh_success_payload_token`) already exercises
`refresh_success_payload(AuthFlow::Token)`. AC-002 may extend that existing test or add a
sibling that also exercises `AuthFlow::OAuth`. The key invariant: this test MUST PASS on the
develop branch before any new code is written (it is a regression pin for already-shipped code).

### AC-006 snapshot placement decision

Keep existing 11 issue snapshots in `src/cli/issue/json_output.rs::#[cfg(test)] mod tests`
(do not move them). Add 4 new auth snapshots in `src/cli/auth.rs::#[cfg(test)] mod tests`.
This maintains cohesion: issue JSON shapes live with issue code, auth JSON shapes live with
auth code. `docs/specs/json-output-shapes.md` cross-references both test locations.

### Snapshot field order

Use `serde_json::json!` literals for all new snapshots (consistent with existing 11 tests).
`json!` macro preserves insertion order, making field order deterministic by construction.
The snapshot-drift risk note from v1.0.0 was overstated — `serde_json::json!` is not
HashMap-based. Only add `Settings::set_sort_maps(true)` if the implementer switches to
`#[derive(Serialize)]` over a struct (not the pattern used here).

### auth.rs size

`src/cli/auth.rs` is approximately 1,997 LOC (NFR-O-D notes it as oversized). This story
makes surgical changes only — add JSON branches to 4 handler functions and thread
`OutputFormat` through their signatures. Do NOT split `auth.rs` in this story — that is
DEFER in Wave 3 (S-3.01). If you find yourself editing more than 80 lines total, stop and
reconsider scope.
