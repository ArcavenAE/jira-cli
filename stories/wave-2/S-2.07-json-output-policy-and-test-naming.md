# S-2.07: JSON output policy decisions (NFR-O-F, NFR-O-J) and test naming convention (NFR-O-W)

## Context

Three MEDIUM NFRs are routed `POLICY-DECISION` in the nfr-catalog.md. This story
resolves all three by making the policy decisions explicit and implementing them:

### NFR-O-F: Auth subcommands missing `--output json` paths

Five auth subcommands (`login`, `switch`, `logout`, `remove`, `refresh`) have no
`--output json` handling. They emit only human-readable output regardless of the
`--output` flag. This violates the general contract (BC-7.3.004) that `--output json`
returns structured JSON for both success and errors.

**Policy decision**: Add JSON output shapes for all five auth subcommands using a
consistent schema:
```json
{ "profile": "<name>", "action": "<verb>", "ok": true }
```
Error shape (to stderr) remains: `{"error": "<message>", "code": <N>}` per BC-7.3.005.

### NFR-O-J: JSON field naming inconsistency

Write operations use four distinct boolean success fields:
- `"changed"` (transitions)
- `"updated"` (issue edit)
- `"linked"` / `"unlinked"` (issue links)

**Policy decision**: The four-distinct-name vocabulary is DELIBERATE verb-aligned naming.
The shape of each is:
- transition: `{"transitioned": bool}` or `{"changed": bool}`
- edit: `{"updated": bool}`
- link: `{"linked": bool}`
- unlink: `{"unlinked": bool}`
- sprint add: `{"added": bool, "sprint_id": N, "issues": [...]}`
- sprint remove: `{"removed": bool, "issues": [...]}`
- auth ops: `{"profile": str, "action": str, "ok": bool}` (NEW from NFR-O-F)

The policy is DOCUMENTED (not changed). Snapshot tests are added to pin the shapes
so a future "harmonization" would be caught.

### NFR-O-W: Mixed test prefix styles

108 tests use `test_<verb>_<subject>` prefix; 212 tests use `<subject>_<verb>_<expected>`
with no prefix. **Policy decision**: Adopt `test_<verb>_<subject>_<expected_outcome>` as
the canonical convention for ALL new tests. Existing tests are NOT renamed (too high
churn). The convention is documented and applied to tests written in this story.

## Behavioral Contracts

**BC-7.3.004**: All commands that support `--output json` return structured JSON on stdout.
Auth subcommands (`login`, `switch`, `logout`, `remove`, `refresh`) must participate.

**BC-7.3.005**: Error outputs when `--output json` is set are structured JSON to stderr:
`{"error": "<message>", "code": <N>}` where `code` matches the exit code.

## Acceptance Criteria

### NFR-O-F: Auth JSON output

**AC-001** (traces to BC-7.3.004 postcondition): Given a profile `default` exists
in config, when `jr auth switch default --output json` is invoked, then exit code is 0
and stdout is `{"profile": "default", "action": "switch", "ok": true}`.

**AC-002** (traces to BC-7.3.004 postcondition): Given a successfully configured auth
profile, when `jr auth refresh --output json` is invoked with a mocked token-refresh
endpoint, then exit code is 0 and stdout is `{"profile": "<name>", "action": "refresh", "ok": true}`.

**AC-003** (traces to BC-7.3.005 postcondition): Given `--output json` is set and an
auth command fails (e.g., `jr auth switch ghost --output json` for non-existent profile),
then exit code is 64 and stderr is parseable JSON with `"error"` and `"code"` keys
where `"code" == 64`. (Extends H-020 coverage to auth subcommands.)

**AC-004** (traces to BC-7.3.004 postcondition): Auth `login --output json` with
`--api-token` flow exits 0 with `{"profile": "<name>", "action": "login", "ok": true}`.
(Tests that the existing login flow now emits JSON when `--output json` is set.)

### NFR-O-J: Verb-aligned naming documentation

**AC-005** (traces to BC-7.3.004 invariant): A new spec file `docs/specs/json-output-shapes.md`
documents the canonical JSON shapes for all write operations. The file includes:
- Transition: `{"transitioned": bool}` (or `{"changed": bool}` — identify which is correct)
- Edit: `{"updated": bool, "key": str}`
- Link/Unlink: `{"linked": bool}` / `{"unlinked": bool}`
- Sprint add/remove: shapes per BC-5.2.007, BC-5.2.008
- Auth ops: new `{"profile": str, "action": str, "ok": bool}` from AC-001..AC-004
- Error: `{"error": str, "code": int}` to stderr

**AC-006** (traces to BC-7.3.004 invariant): Snapshot tests for at least 3 existing
write operation JSON shapes are added (using `insta` snapshot testing) to prevent
silent changes to existing shapes. New tests follow the `test_<verb>_<subject>_<expected>`
naming convention.

### NFR-O-W: Test naming convention

**AC-007** (traces to BC-6.1.001 invariant): CLAUDE.md Conventions section contains:
```
**Test naming:** New tests use `test_<verb>_<subject>_<expected_outcome>` convention
(e.g., `test_auth_switch_returns_json_ok`). Existing tests with no-prefix names are
NOT renamed; this convention applies to new tests only.
```

**AC-008** (traces to BC-6.1.001 invariant): `docs/specs/test-naming-convention.md`
documents the convention rationale, examples, and the migration policy (new tests only).

**AC-009** (traces to BC-6.1.001 invariant): All test functions written in this story
(S-2.07) follow the `test_<verb>_<subject>_<expected_outcome>` naming convention.

## Implementation Notes

### NFR-O-F: Implementing JSON output for auth subcommands

In `src/cli/auth.rs`, each auth subcommand handler needs an `--output json` branch:

```rust
// Pattern: at the end of each handler, after success:
if output == OutputMode::Json {
    let json = serde_json::json!({
        "profile": profile_name,
        "action": "switch",  // or "login", "logout", "remove", "refresh"
        "ok": true
    });
    println!("{}", json);
} else {
    eprintln!("Switched to profile '{}'", profile_name);
}
```

Note: `src/cli/auth.rs` is 1,997 LOC (NFR-O-D notes it as oversized). This story
modifies the existing file without splitting it (splitting is DEFER in Wave 3). Make
surgical changes only — add the JSON branch to each of the 5 subcommand handlers
without restructuring the file.

### NFR-O-J: Identifying exact field names

Before writing the spec, verify the ACTUAL field names used by examining the source:
- `src/cli/issue/workflow.rs`: what does `issue move` emit on `--output json`?
- `src/cli/issue/create.rs`: what does `issue edit` emit?
- `src/cli/issue/links.rs`: what do `issue link`/`issue unlink` emit?

Document what they actually emit, not what they should emit. The policy is to
PRESERVE these names (verb-aligned), not change them.

### Snapshot tests (AC-006)

Use `insta::assert_json_snapshot!` for at least 3 shapes. Test names follow
the new convention:
```rust
#[test]
fn test_issue_move_returns_transitioned_false_json() { ... }
```

## Architecture Compliance Rules

- `--output` flag check must use the existing `OutputMode` enum pattern (as in
  `issue list`, `issue view`, etc.). Do not introduce a new enum.
- Auth JSON output must go to stdout on success, stderr on error — per BC-7.3.004
  and BC-7.3.005.
- The `{"profile": str, "action": str, "ok": bool}` shape is NEW (not previously used).
  It must be documented in `docs/specs/json-output-shapes.md` (AC-005).
- Snapshot tests use `insta` — already in dev-deps.
- Reference: `architecture/module-decomposition.md` for auth.rs and output.rs.

## Risk / Dependencies

- **No Wave 0 or Wave 1 dependencies.** Independent of all other Wave 2 stories.
- **Risk: auth.rs size.** At 1,997 LOC, loading the full file into agent context is
  expensive. Load only the relevant handler functions (login, switch, logout, remove,
  refresh). Use the Architecture section of CLAUDE.md to locate function names first.
- **Risk: snapshot drift.** If AC-006 snapshot tests capture slightly different output
  than expected (e.g., due to environment-specific ordering), the snapshots will fail
  on CI. Use `insta::assert_json_snapshot!` with explicit `{}` sorting if field order
  is non-deterministic.
- **Risk: `issue move` field name.** The actual field name for move output (`transitioned`
  vs `changed`) must be verified in source before documenting. Do not assume.
- **No breaking change.** Auth subcommands with `--output json` are additive. Existing
  human-readable output is unchanged when `--output json` is NOT set.
